\chapter{VR-Calibrated Motion--Force Demonstration Collection}
\label{chap:demonstration_collection}

\section{Teaching-System Purpose}
The demonstration system must capture how a human moves the polishing tool and how the tool interacts with the surface. FAIRE uses a handheld teaching device instrumented with a six-axis force/torque sensor. A VR tracker measures the device pose, and an RGB camera records visual task context. These streams are synchronized so that each visual and kinematic state is paired with the wrench produced at the same stage of contact.

The device is intentionally separated from the robot. This arrangement enables natural task demonstration in the workpiece area, but it creates a calibration problem: tracker poses are measured in a VR tracking frame, whereas the learned trajectory must be executed in the robot base frame.

\section{Coordinate Frames and Pose Transfer}
Let \(B\), \(V\), \(T\), and \(E\) denote the robot base, VR tracking, tracker, and end-effector/tool frames. The demonstrated tool pose expressed in the robot base frame is
\begin{equation}
  {}^{B}\!T_{E}^{\mathrm{demo}}(t)
  =
  {}^{B}\!T_{V}\,
  {}^{V}\!T_{T}(t)\,
  {}^{T}\!T_{E}.
  \label{eq:demo_transfer}
\end{equation}
The time-varying transform \({}^{V}\!T_T(t)\) is measured by the tracking system. The constant transform \({}^{T}\!T_E\) describes the mechanical offset from tracker to tool, and \({}^{B}\!T_V\) aligns the VR tracking coordinates with the robot base.

Equation~\eqref{eq:demo_transfer} makes the transfer convention explicit. It also exposes the calibration components that must be validated separately. Errors in \({}^{B}\!T_V\) shift or rotate the entire demonstration in the robot workspace, whereas errors in \({}^{T}\!T_E\) produce a tool-dependent offset. The final implementation should report the calibration observations, estimator, residual translational and rotational errors, and validation procedure. These details are \todoexp{not specified in the source FAIRE manuscript}.

\section{Wrench Processing}
The force/torque sensor measures
\begin{equation}
  w_t=[F_x,F_y,F_z,\tau_x,\tau_y,\tau_z]_t.
\end{equation}
Raw readings require bias compensation before they are used for learning or control. The compensated wrench must then be represented in a consistent frame. If the sensor-frame wrench is transformed to the tool or robot frame, both force and torque must follow the appropriate spatial wrench transformation; simply rotating the force vector is insufficient when the reference point changes.

The FAIRE manuscript states that bias compensation and frame normalization are performed but does not define the calibration pose, averaging interval, gravity/tool-load compensation, reference point, or filter. The thesis therefore treats these as required implementation records rather than assigning unverified values. The same convention must be used for demonstration targets, policy observations, measured execution forces, and evaluation metrics.

\section{Synchronization and Dataset Construction}
After calibration and preprocessing, a demonstration episode is stored as
\begin{equation}
  D=\{I_t,x_t,R_t,w_t\}_{t=1}^{T},
  \label{eq:demonstration_dataset}
\end{equation}
where \(x_t\) and \(R_t\) are the tool position and orientation expressed in the robot workspace. A usable dataset also requires a timestamp for each stream, episode boundaries, and enough metadata to reproduce the coordinate convention and preprocessing.

The final synchronization method is \todoexp{TBD}. It should document whether measurements are hardware-triggered, timestamp-aligned, or resampled; the image, tracker, wrench, and controller rates; interpolation rules; the maximum accepted skew; and the treatment of missing samples. Temporal alignment is especially important because the learning target contains both motion and force. A systematic delay can make the policy associate a pose with a force that was generated earlier or later in the stroke.

\section{Segmentation and Training Windows}
Continuous episodes are divided into history-and-target samples. At time \(t\), the input includes current visual and pose information and the wrench history \(w_{t-K:t}\). The target contains a future horizon of reference positions, orientations, and desired forces from \(t\) through \(t+H\). Samples near episode boundaries require either shortened windows, padding, or exclusion; the selected rule must be shared by all ablation variants.

Train, validation, and test partitions should be defined at the episode or workpiece-trial level rather than by randomly mixing adjacent windows. Otherwise, nearly identical temporal neighbors can leak across partitions. The split, number of demonstrations, window stride, \(K\), \(H\), normalization statistics, and augmentation policy remain \todoexp{TBD pending the final dataset}.

\section{Demonstration Quality and Safety}
The collection protocol should reject or label demonstrations with sensor saturation, tracking loss, unsafe load, accidental contact, missing images, or timestamp discontinuities. The robot must not replay a transferred demonstration until the transformed trajectory has passed workspace, reachability, collision, velocity, and force-limit checks. Calibration enables coordinate transfer; it does not by itself establish safe executability.

For reproducibility, each accepted episode should include the workpiece identifier, target region, sensor zeroing state, calibration version, tool configuration, start pose, sampling metadata, and operator annotation. These fields are listed in Appendix~\ref{chap:appendix}.
