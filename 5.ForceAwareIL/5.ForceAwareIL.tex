\chapter{Force-History Imitation and Admittance Execution}
\label{chap:force_il}

\section{Force-Aware Observation}
The imitation policy receives complementary visual, kinematic, and interaction observations. The image \(I_t\) describes the specified task region, the robot state \(s_t\) describes the current tool pose and proprioception, and the wrench history \(w_{t-K:t}\) describes recent contact evolution. Vision is not used as an autonomous defect detector in this thesis.

Separate encoders compute
\begin{align}
  z_t^I &= f_I(I_t),\\
  z_t^P &= f_P(s_t),\\
  z_t^F &= \mathrm{GRU}(w_{t-K:t}).
  \label{eq:policy_encoders}
\end{align}
The force branch uses the full six-axis wrench rather than normal force alone so that tangential force and torque can provide evidence about sliding, tool tilt, or partial contact. The exact image backbone, state variables, layer dimensions, and GRU hidden size are \todoexp{TBD}.

\section{Motion--Force Trajectory Prediction}
The fused representation is mapped to a short reference horizon:
\begin{equation}
  \pi_\theta(z_t)\rightarrow
  \left\{
    x^{\mathrm{ref}}_{t:t+H},
    R^{\mathrm{ref}}_{t:t+H},
    F^{\mathrm{des}}_{t:t+H}
  \right\}.
  \label{eq:motion_force_prediction}
\end{equation}
Position and orientation define the demonstrated motion, while \(F^{\mathrm{des}}\) defines the demonstrated contact-force behavior to be realized by the controller. In the experimental notation of Chapter~\ref{chap:experiments}, the six-dimensional motion representation and three-dimensional force form a 9D action at each horizon step.

Predicting a horizon rather than a single next sample provides a reference segment that can be executed smoothly and supports the planned comparison between one-step force prediction and force-trajectory prediction. The source manuscript does not specify how overlapping horizons are combined online. Action selection, temporal ensembling, replanning frequency, and horizon interruption are therefore \todoexp{TBD}.

\section{Supervised Learning Objective}
Training samples are drawn from the calibrated demonstration dataset in Eq.~\eqref{eq:demonstration_dataset}. The multitask objective is
\begin{equation}
  \mathcal{L}_{\mathrm{IL}}
  =\lambda_x\mathcal{L}_x
  +\lambda_R\mathcal{L}_R
  +\lambda_F\mathcal{L}_F,
  \label{eq:imitation_loss}
\end{equation}
where \(\mathcal{L}_x\), \(\mathcal{L}_R\), and \(\mathcal{L}_F\) penalize position, orientation, and force-trajectory prediction error. Position and force can be trained with component-wise or norm-based regression losses. Orientation requires a representation-aware error, such as a geodesic rotation loss, if the output is a rotation matrix or quaternion.

The source manuscript deliberately leaves the exact loss definitions and weights open. The final thesis must report output normalization, orientation parameterization, force channels, masking of padded targets, optimizer, learning-rate schedule, batch size, epoch or step count, seed count, checkpoint selection, and regularization. Until those records are available, Eq.~\eqref{eq:imitation_loss} is a framework-level specification rather than a complete reproduction recipe.

\section{Why Force History Is Separated}
A unified encoder can learn correlations among pose and force, but it does not explicitly preserve the distinct temporal structure of contact. FAIRE separates the force branch so that its representation can be ablated independently. The experimental comparison includes a unified current-sample MLP, split current-sample MLP, split history MLP, split history LSTM, and split history GRU. Each variant retains the same 9D trajectory output and admittance execution, isolating the effect of force representation.

The claim tested by this comparison is that temporal force conditioning improves contact reproduction. The GRU architecture itself is not claimed as a new recurrent model.

\section{Admittance-Based Force-Trajectory Execution}
\label{sec:admittance_execution}
The low-level controller converts desired force into a Cartesian correction according to
\begin{equation}
  M_d\Delta\ddot{x}_t+
  D_d\Delta\dot{x}_t+
  K_d\Delta x_t
  =
  F_t^{\mathrm{meas}}-F_t^{\mathrm{des}},
  \label{eq:admittance}
\end{equation}
where \(M_d\), \(D_d\), and \(K_d\) are virtual inertia, damping, and stiffness matrices in the controlled contact frame. The corrected position command is
\begin{equation}
  x_t^{\mathrm{cmd}}=x_t^{\mathrm{ref}}+\Delta x_t.
  \label{eq:position_command}
\end{equation}

This equation defines the semantics of the learned force output. The policy predicts a reference force; the controller observes the tracking error and moves compliantly to reduce it. The imitation policy is not itself an admittance controller, and the force value is not directly mapped to actuator torque.

\section{Discrete Execution and Safety Requirements}
Implementation requires discretizing Eq.~\eqref{eq:admittance} at the controller rate, filtering the measured wrench, selecting controlled axes, and limiting \(\Delta x_t\), \(\Delta\dot{x}_t\), and total commanded motion. Contact establishment and release also require explicit state handling so that the controller does not integrate noise while the tool is away from the surface.

The final controller specification must include \(M_d\), \(D_d\), \(K_d\), update rate, filter, force frame, saturation values, contact threshold, anti-windup or state-reset rule, and robot command interface. These quantities are \todoexp{TBD}; no claim of robustness to varying or learned admittance mass and damping is made.

\section{Online Execution Sequence}
At each policy update, FAIRE performs the following conceptual sequence:
\begin{enumerate}
  \item acquire and preprocess the current image, robot state, and wrench;
  \item update the wrench-history buffer and compute \(z_t^F\);
  \item predict a motion--force reference horizon;
  \item select the current reference sample and compute force error;
  \item integrate the admittance dynamics subject to safety limits;
  \item apply the residual velocity and orientation corrections defined in Chapter~\ref{chap:residual_rl};
  \item transmit the resulting Cartesian command through the robot's safe command interface.
\end{enumerate}
Policy and controller updates may operate at different rates. Their exact scheduling is part of the pending implementation record.
